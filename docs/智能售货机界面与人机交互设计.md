# 智能售货机界面与人机交互设计

## 1. 设计目标

本界面面向 QSM368ZP-WF 开发板智能零售赛题演示，目标是在比赛现场清楚展示“摄像头识别、端侧处理、购物车计价、语音交互、支付闭环”五个能力。

界面设计遵循三个原则：

- 第一眼能看懂：评委看到屏幕即可理解系统正在识别商品并计算金额。
- 操作路径短：扫码、语音询问、查看购物车、结算都能在同一界面完成。
- 适合板端性能：优先使用 LVDS 上的实时摄像头画面叠加文字，不依赖复杂动画和重型 UI。

## 2. 硬件与交互位置

推荐实体布局如下：

```text
┌──────────────────────────────────────────────┐
│                 LVDS 显示屏                   │
│  左侧：摄像头实时画面      右侧：购物车/语音状态 │
└──────────────────────────────────────────────┘

          ↑ 屏幕下方或侧边固定摄像头
          │ 摄像头朝向商品扫码区

┌──────────────────────┐   ┌──────────────────┐
│ 商品/二维码放置区      │   │ 麦克风/语音交互区  │
│ 距离摄像头 15-30 cm   │   │ 靠近用户正前方     │
└──────────────────────┘   └──────────────────┘

┌──────────────────────────────────────────────┐
│  电脑终端/ADB 调试区：仅用于开发和备用演示       │
└──────────────────────────────────────────────┘
```

### 摄像头位置

- 摄像头固定在 LVDS 屏幕下方或左侧，朝向商品放置区。
- 商品二维码建议距离摄像头 15-30 cm。
- 摄像头画面中二维码占画面约 1/4，便于稳定识别。
- 商品放置区应有明显边界，例如贴纸标注“扫码区”。

### 麦克风位置

- 麦克风放在用户正前方 20-50 cm 范围内。
- 麦克风旁边标注“可询问：牛奶多少钱、还有什么商品、当前购物车、结算”。
- 如果现场环境嘈杂，建议使用固定短句演示，减少 ASR 误识别。

### 终端位置

- 终端作为开发调试和备用显示，不应成为主交互界面。
- 比赛现场优先让评委看 LVDS 屏幕。
- 终端保留用于启动程序、查看日志、处理异常。

## 3. LVDS 主界面布局

推荐采用左右分栏布局，比例约为 60% : 40%。

```text
┌──────────────────────────────┬──────────────────────┐
│ 摄像头实时画面                │ 系统状态              │
│                              │ 当前模式：扫码识别     │
│  ┌──────────────────────┐    │ 网络：Wi-Fi 已连接     │
│  │ 绿色二维码识别框       │    │ 语音：监听中           │
│  └──────────────────────┘    ├──────────────────────┤
│                              │ 当前商品              │
│  底部叠加：识别结果/置信度     │ 名称：Milk            │
│                              │ 单价：CNY 4.50        │
├──────────────────────────────┼──────────────────────┤
│ 操作提示                      │ 购物车                │
│ 请将商品二维码放入框内          │ Milk x1  CNY 4.50     │
│ 可说：牛奶多少钱 / 结算         │ Cola x1  CNY 3.50     │
│                              │ 总价：CNY 8.00        │
└──────────────────────────────┴──────────────────────┘
```

### 左侧摄像头画面区

显示内容：

- 摄像头实时画面。
- 二维码或商品识别框。
- 当前识别状态，例如 `Scanning...`、`QR detected`、`Added Milk`。
- 若后续接入 RKNN 商品识别，可显示 `AI: Milk 92%`。

作用：

- 证明摄像头链路真实运行。
- 展示端侧识别结果不是纯终端模拟。
- 让评委直观看到商品进入识别区域。

### 右侧状态与购物车区

显示内容：

- 网络状态：Wi-Fi connected / offline。
- 语音状态：Listening / Recognized / Speaking。
- 当前商品名称、单价、加入状态。
- 购物车列表、数量、总价。
- 支付链接或支付二维码占位。

作用：

- 展示从识别到计价的业务闭环。
- 让用户不需要看终端也能知道当前订单状态。

## 4. 终端显示设计

终端用于后台和调试，建议保持简洁。

推荐输出格式：

```text
[CAMERA] QR detected: product:milk
[CART] Added Milk, price CNY 4.50, total CNY 4.50
[VOICE] Recognized: 牛奶多少钱
[ANSWER] Milk price is CNY 4.50.
[PAY] Order created: QSM202607070001
```

终端不建议承担主视觉展示，因为：

- 字体小，评委距离稍远时不易看清。
- 中文编码可能出现乱码。
- 终端无法清楚展示摄像头画面和购物车状态。

终端适合保留为“后台日志 + 应急控制台”。

## 5. 人机交互流程

### 流程一：扫码购物

```text
用户将商品二维码放到扫码区
-> 摄像头画面出现绿色识别框
-> 系统识别商品 ID
-> 商品自动加入购物车
-> LVDS 显示商品名、单价、总价
```

界面反馈：

- 摄像头区显示绿色框。
- 当前商品区显示 `Added Milk`。
- 购物车区数量加 1。
- 终端输出识别日志。

### 流程二：语音问答

```text
用户说：牛奶多少钱
-> 麦克风录音
-> ASR 识别文本
-> 本地商品规则或豆包模型生成回答
-> 终端显示回答
-> 后续 TTS 播放中文语音
```

界面反馈：

- 语音状态显示 `Listening...`。
- 识别后显示 `Recognized`。
- 回答区显示价格或智能回复。

### 流程三：查看商品

```text
用户说：还有什么商品
-> 系统读取 products.csv
-> 显示商品列表
```

界面反馈：

- 右侧商品列表滚动或分页显示。
- 终端同步输出商品名称和价格。

### 流程四：结算支付

```text
用户说：结算
-> 系统生成订单号
-> 根据购物车计算总价
-> 生成支付链接或支付二维码
-> 屏幕显示订单金额和支付二维码
```

界面反馈：

- 总价区域高亮。
- 支付区显示二维码或链接。
- 若支付接口未接入，显示“演示支付链接”。

## 6. 信息层级设计

屏幕信息优先级从高到低：

1. 当前识别商品。
2. 购物车总价。
3. 摄像头识别框。
4. 语音状态。
5. 商品列表和支付信息。
6. 调试日志。

不建议在 LVDS 上显示过多日志。日志应放在终端，屏幕只显示用户需要理解的状态。

## 7. 推荐颜色与状态

| 状态 | 颜色 | 含义 |
| --- | --- | --- |
| 绿色 | 识别成功、已加入购物车 | 成功反馈 |
| 黄色 | 正在识别、正在联网、正在语音识别 | 等待反馈 |
| 红色 | 未识别、网络失败、支付失败 | 错误反馈 |
| 白色 | 商品名、价格、普通信息 | 主文本 |
| 深灰 | 背景和分隔区域 | 降低视觉干扰 |

## 8. 适合当前项目的实现路线

### 第一阶段：在现有 DRM 画面上叠加文字

当前项目已有 `qr_scanner_display`，可以优先在该程序中叠加：

- 当前识别商品名。
- 商品价格。
- 购物车总价。
- 语音状态。

优点：

- 改动最小。
- 直接复用摄像头实时画面。
- 最适合短期比赛演示。

### 第二阶段：业务逻辑与扫码程序合并

将二维码识别结果直接传给购物车逻辑：

```text
QR result -> product id -> Catalog -> Cart -> UI overlay
```

完成后，系统就能从“扫码显示”升级为“扫码购物”。

### 第三阶段：增加更完整 UI

如果时间充足，可以考虑：

- LVGL：适合嵌入式控件和触控界面。
- Qt：适合完整桌面式界面，但资源占用更高。
- HTML 本地页面：适合快速做漂亮界面，但需要浏览器或 WebView 支持。

当前阶段建议优先走 DRM 叠字，不建议一开始重做复杂 GUI。

## 9. 比赛现场推荐演示脚本

1. 开机后启动 Wi-Fi 和语音问答脚本。
2. 启动 LVDS 摄像头识别界面。
3. 把商品二维码放到摄像头前。
4. 屏幕出现绿色框并显示商品名。
5. 商品加入购物车，总价变化。
6. 对麦克风说“牛奶多少钱”。
7. 终端或屏幕显示价格回答。
8. 对麦克风说“还有什么商品”。
9. 系统显示商品列表。
10. 对麦克风说“结算”。
11. 屏幕显示订单号、总价和支付链接。
12. 讲解后续 RKNN 商品识别扩展。

## 10. 风险与应急方案

| 风险 | 现场表现 | 应急方案 |
| --- | --- | --- |
| 二维码识别不稳定 | 绿色框不出现 | 增大二维码、改善光线、调整距离 |
| ASR 识别不准 | 语音问题无回复 | 使用终端输入 `ask` 作为备用 |
| 中文终端乱码 | 识别文本显示异常 | LVDS 和终端回答使用英文或图形状态 |
| TTS 未授权 | 无语音播报 | 保留屏幕文字回答 |
| Wi-Fi 断开 | 云端 ASR/LLM 不可用 | 运行 `scripts/connect_wifi.sh` 重连 |
| 摄像头被占用 | 画面无法打开 | 停止旧进程后重启扫码程序 |

## 11. 最终推荐界面方案

最终建议采用：

```text
LVDS 主屏 = 摄像头实时画面 + 商品识别框 + 购物车状态
终端 = 后台日志 + 应急输入
麦克风 = 用户自然语言入口
扬声器 = 中文语音播报出口
摄像头 = 商品/二维码识别入口
```

这个方案最符合赛题要求，因为它同时体现：

- 端侧摄像头采集。
- 端侧二维码/商品识别。
- 智能购物车计价。
- 语音人机交互。
- 云端 ASR/LLM/TTS 扩展。
- 后续 RKNN 端侧 AI 升级空间。
