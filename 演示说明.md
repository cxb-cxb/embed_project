# 10 类商品视觉识别 + LVDS 叠加显示演示说明

## 当前能力

本演示用于 QSM368ZP-WF 开发板上的智能零售原型展示：

- 从板端摄像头 `/dev/video5` 抓取实时画面。
- 对 10 类商品做视觉识别。
- 未录入商品会显示为 `unknown / 未录入商品`，不会强行归到 10 类商品中。
- 在 LVDS 屏幕上显示摄像头画面、识别框、商品名和置信度。
- 当前已实测牛奶识别稳定，10 轮结果均为 `milk / 牛奶`，置信度 `0.990`。

支持的 10 类商品：

| 英文 ID | 中文名 |
|---|---|
| cola | 可乐 |
| noodle | 方便面 |
| chips | 薯片 |
| biscuit | 饼干 |
| milk | 牛奶 |
| bread | 面包 |
| toothpaste | 牙膏 |
| water | 矿泉水 |
| tissue | 纸巾 |
| soap | 肥皂 |

## 一键启动

在 Windows 资源管理器中进入本目录：

```text
C:\Users\33591\Documents\嵌赛 2
```

双击：

```text
start_visual_lvds_demo.cmd
```

默认会持续运行演示，按 `Ctrl+C` 停止。

当前版本默认使用连续摄像头流，不再每轮重新抓一张照片，因此 LVDS 画面会持续刷新。识别结果默认每 5 帧更新一次，画面每帧刷新一次。

## 有限轮次测试

如果只想跑固定次数，例如 10 轮：

```powershell
.\start_visual_lvds_demo.ps1 -Rounds 10 -Interval 0.5
```

如果需要更频繁更新识别结果：

```powershell
.\start_visual_lvds_demo.ps1 -ClassifyEvery 2
```

如果连续流异常，可以回退到旧的单帧抓拍模式：

```powershell
.\start_visual_lvds_demo.ps1 -SingleShot
```

如果已经确认环境正常，想跳过预检查：

```powershell
.\start_visual_lvds_demo.ps1 -Rounds 10 -Interval 0.5 -SkipChecks
```

## 演示前检查

启动前确认：

- QSM368ZP-WF 开发板已开机。
- USB 已连接电脑。
- 板子处于正常 ADB 模式，不是 Loader 模式。
- LVDS 屏幕已接好并点亮。
- 摄像头对准待识别商品，商品尽量正对画面。

脚本会自动检查：

- Python 依赖：`opencv-python`、`numpy`、`pillow`
- ADB 在线设备
- LVDS 状态
- Python 脚本语法

## 常见问题

### ADB 找不到设备

先检查：

```powershell
adb devices -l
```

正常应看到：

```text
device product:rk3568-linux model:Nexus_4 device:mako
```

如果没有设备，重新插拔 USB，确认开发板不是 Loader 模式。

### LVDS 未连接

脚本会读取：

```text
/sys/class/drm/card0-LVDS-1/status
```

正常结果应为：

```text
connected
```

如果不是 `connected`，检查屏线、电源、背光和 Weston 状态。

### 识别结果不稳定

优先调整：

- 商品放在画面中央。
- 商品正面对摄像头。
- 避免手遮挡包装文字。
- 保持背景尽量简单。
- 不要离摄像头太近或太远。

当前算法使用 ORB 包装关键点匹配，对包装文字和图案比较敏感。若更换商品包装，需要重新采集样本。

### 未录入商品误判为已录入商品

当前版本已加入未知商品判定：

- 有效关键点匹配数量不足时显示 `未录入商品 / unknown`。
- 前几名匹配结果分散时显示 `未录入商品 / unknown`。
- 已录入商品仍会显示具体商品名和置信度。

如果某个未录入商品仍被误判，说明它的包装图案和已录入商品过于相似，需要提高 `tools/live_lvds_product_overlay.py` 中的 `MIN_ORB_GOOD_MATCHES` 阈值，或采集更多负样本做专门的未知类判断。

Demos updated note - LVDS realtime preview mode
----------------------------------------------

Default command:

```powershell
.\start_visual_lvds_demo.ps1
```

The default `DisplayMode` is now `preview`.

- The board runs a GStreamer `tee` pipeline.
- One branch sends the camera stream directly to LVDS at realtime speed.
- Another branch writes small JPEG preview frames under `/tmp/live_product_preview_*.jpg`.
- The PC pulls those JPEG frames at the configured interval for product recognition.

This avoids the old slow path of pulling a full camera frame to the PC and pushing a full overlay image back to LVDS for every frame.

Fallback overlay modes:

```powershell
.\start_visual_lvds_demo.ps1 -DisplayMode fb
.\start_visual_lvds_demo.ps1 -DisplayMode kms
```

These modes keep the old visual overlay behavior, but they are slower and should only be used when the overlay itself is required.
